## 열거형 정의

### `BizStatus` — 거래 상태코드

| 값               | 설명                                                                                                                                                                                                         |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `INIT`          | **거래 전문이 최초로 시스템에 수신된 상태**를 의미한다. 아직 실제 비즈니스 처리가 시작되지 않은 초기 단계이며, 전문 파싱 및 기본 검증(필수 필드 확인, 형식 검증 등)이 수행되는 단계이다. 일반적으로 거래 레코드를 생성할 때 최초 상태로 설정될 수 있으며 이후 처리 흐름에 따라 `PROCESSING`, `NORMAL`, `ERROR` 등으로 전이된다. |
| `NORMAL`        | **거래가 정상적으로 완료된 상태**이다. 입금, 출금, 조회 등의 비즈니스 로직이 성공적으로 수행되었음을 의미한다. 이 상태가 되면 해당 거래는 최종 완료 상태로 간주되며 동일 거래 식별자의 전문이 다시 수신될 경우 보통 **중복 거래 처리 정책**에 따라 `DROP` 또는 `REPLY_ORIGINAL` 처리가 수행된다.                      |
| `ERROR`         | **거래 처리 중 오류가 발생하여 실패한 상태**이다. 계좌 검증 실패, 잔액 부족, 시스템 오류, 외부 기관 응답 오류 등의 다양한 원인으로 발생할 수 있다. 이 상태에서는 실제 금전 처리가 완료되지 않았을 가능성이 높으며, 동일 거래가 재수신될 경우 정책에 따라 기존 오류 응답을 그대로 반환하거나 재처리를 제한한다.                        |
| `PROCESSING`    | **거래가 현재 처리 중인 상태**이다. 거래가 시작되어 아직 최종 결과가 확정되지 않은 단계이며, 비즈니스 로직이 실행 중이거나 외부 시스템 응답을 기다리는 상태일 수 있다. 이 상태에서 동일 거래 전문이 재수신될 경우 대부분의 경우 **중복 처리 방지를 위해 `DROP` 처리**된다.                                          |
| `CANCEL_NORMAL` | **취소 거래가 정상적으로 완료된 상태**이다. 기존에 처리된 거래(입금 또는 출금 등)에 대해 취소 요청이 수행되어 금액 복구 및 거래 취소 기록 저장이 정상적으로 완료된 경우이다. 이 상태에서는 동일 취소 전문이 재수신되면 보통 기존 취소 결과를 그대로 반환(`REPLY_ORIGINAL`)하여 **취소 거래의 멱등성**을 유지한다.               |
| `CANCEL_ERROR`  | **취소 처리 중 오류가 발생한 상태**이다. 취소 과정에서 시스템 오류, 외부 기관 오류, 데이터 정합성 문제 등이 발생하여 취소가 정상적으로 완료되지 못한 경우이다. 이 상태에서는 동일 취소 전문이 재수신되면 정책에 따라 **취소 재처리(PROCEED)** 가 허용될 수 있다.                                              |
| `CANCEL_PROC`   | **취소 거래가 현재 처리 중인 상태**이다. 취소 요청이 접수되어 취소 로직(금액 복구, 거래 취소 기록 저장 등)이 진행 중인 단계이다. 이 상태에서 동일 취소 전문이 다시 수신되면 중복 취소 처리나 동시성 문제가 발생할 수 있으므로 대부분 **전문을 무시(DROP)** 하여 기존 취소 처리 완료를 기다리도록 한다.                        |


### `ProcessResult` — 처리 결과 지시자

| 값                 | 설명                                                                                                                                                                                                                                                                  |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `PROCEED`         | **정상 프로세스 진행**을 의미한다. 해당 전문이 신규 거래이거나 정상적으로 처리해야 하는 상황일 때 반환된다. 시스템은 이후 비즈니스 로직을 수행하며 계좌 검증, 금액 처리, 거래 기록 저장 등의 절차를 진행한다. 일반적으로 처리 시작 시 거래 상태를 `PROCESSING`으로 설정하고, 처리 완료 후 결과에 따라 `NORMAL` 또는 `ERROR` 상태로 업데이트한다.                                                  |
| `DROP`            | **중복 또는 불필요한 전문을 무시**해야 하는 경우이다. 동일 거래 식별자(거래번호, 전문번호 등)를 가진 거래가 이미 처리되었거나 현재 처리 중인 경우에 발생한다. 금융 전문 통신 환경에서는 네트워크 지연이나 응답 미수신으로 인해 동일 전문이 재전송될 수 있으므로, 이러한 중복 전문을 그대로 처리하면 중복 입금·출금 등의 문제가 발생할 수 있다. 따라서 시스템은 해당 전문을 처리하지 않고 무시하며, 필요 시 로그만 기록한다.                   |
| `REPLY_ORIGINAL`  | **기존 원거래의 처리 결과를 그대로 응답**해야 하는 상황이다. 동일 거래에 대한 전문이 재수신되었을 때 신규 처리 대신 기존 응답 데이터를 그대로 반환한다. 예를 들어 이미 정상 처리된 입금 또는 이미 완료된 취소 요청이 다시 수신된 경우이다. 이를 통해 거래 처리의 **멱등성(Idempotency)** 을 보장하고 동일 거래가 여러 번 처리되는 것을 방지한다.                                                       |
| `ERROR_NOT_FOUND` | 취소 요청 등에서 **참조해야 할 원거래가 시스템에 존재하지 않는 경우** 발생하는 오류 처리 결과이다. 정상적인 금융 거래 흐름에서는 취소 요청 이전에 반드시 원거래가 존재해야 하므로, 이러한 상황은 비정상 상태로 판단된다. 시스템은 오류 응답을 반환하고 로그 및 모니터링을 통해 운영자가 확인할 수 있도록 한다.                                                                                    |
| `SAVE_REVERSE`    | **역전 거래(Reverse Transaction) 상황에 대비하여 전문을 우선 저장**해야 하는 경우이다. 일반적으로 취소 요청에 해당하는 원거래가 시스템에 존재하지 않을 때 발생한다. 외부 기관에서는 이미 취소 처리가 완료되었지만 내부 시스템에는 원거래 기록이 없는 경우가 있을 수 있다. 이때 시스템은 취소 전문을 우선 저장하여 거래 이력을 보존하고 이후 정합성 검증 또는 후속 처리에 활용할 수 있도록 한다. 이러한 상황은 보통 운영 모니터링 대상이 된다. |

## 거래 유형별 처리 기준

### 1. 조회 (INQUIRY) 처리 기준

```java
/**
 * 1. 조회 (INQUIRY) 처리 기준
 *
 * [원거래 존재 + 상태 NORMAL / ERROR / PROCESSING]
 * - 동일한 거래 식별자(거래번호, 전문번호 등)를 가진 조회 요청이 이미 처리되었거나 처리 중인 상태이다.
 * - 조회(INQUIRY)는 계좌 상태나 금액을 변경하지 않는 단순 조회성 트랜잭션이다.
 * - 금융 전문 통신 환경에서는 응답 지연, 네트워크 재전송 등으로 동일 전문이 반복 수신될 수 있다.
 * - 이 경우 시스템은 중복 요청으로 판단하고 추가 처리 없이 전문을 무시(DROP)한다.
 * - 기존 거래 결과를 신뢰하여 불필요한 DB 조회 및 외부 기관 연동을 방지한다.
 * - 이를 통해 조회 트랜잭션의 멱등성(Idempotency)을 보장한다.
 *
 * [원거래 미존재]
 * - 시스템에 동일 거래 식별자가 존재하지 않는 경우 최초 조회 요청으로 판단한다.
 * - 정상적인 조회 프로세스를 수행한다.
 *   1) 거래 정보 저장 또는 조회 로그 기록
 *   2) DB 조회 또는 외부 기관 조회 수행
 * - 처리 완료 후 결과에 따라 거래 상태를 기록한다.
 *   - 정상 처리: NORMAL
 *   - 오류 발생: ERROR
 */
```

### 2. 입금 (DEPOSIT) 처리 기준

```java
/**
 * 2. 입금 (DEPOSIT) 처리 기준
 *
 * [원거래 존재 + 상태 NORMAL]
 * - 이미 정상 처리된 입금 거래가 존재하는 경우이다.
 * - 동일 거래번호(거래고유번호, 전문번호 등)의 입금 전문이 재수신되면 중복 거래로 판단한다.
 * - 금융 전문 통신 환경에서는 네트워크 지연, 응답 미수신 등으로 동일 전문이 재전송될 수 있다.
 * - 이 경우 시스템은 기존에 저장된 원거래 응답 데이터를 그대로 재응답(REPLY_ORIGINAL)한다.
 * - 이를 통해 중복 입금 처리 방지 및 멱등성(Idempotency)을 보장한다.
 *
 * [원거래 존재 + 상태 ERROR]
 * - 이전 입금 거래가 오류(ERROR) 상태로 종료된 경우이다.
 * - 동일 전문이 재수신되더라도 신규 거래로 처리하지 않는다.
 * - 기존 오류 응답을 그대로 반환(REPLY_ORIGINAL)한다.
 * - 이는 동일 거래에 대한 반복 재처리를 방지하고 거래 결과의 일관성을 유지하기 위한 정책이다.
 *
 * [원거래 존재 + 상태 PROCESSING]
 * - 동일 거래가 현재 처리 중인 상태이다.
 * - 동일 전문이 추가 수신되면 병렬 처리로 인해 중복 입금 또는 데이터 충돌이 발생할 수 있다.
 * - 따라서 해당 전문은 무시(DROP)하고 현재 진행 중인 거래가 완료될 때까지 추가 처리를 하지 않는다.
 * - 필요 시 로그만 기록한다.
 *
 * [원거래 미존재]
 * - 동일 거래 식별자를 가진 원거래가 시스템에 존재하지 않는 경우 신규 입금 요청으로 판단한다.
 * - 정상적인 입금 프로세스를 수행한다.
 *
 * 처리 절차
 * 1) 거래 정보 생성 및 상태 PROCESSING 기록
 * 2) 계좌 검증 및 금액 처리
 * 3) 거래 기록 저장
 *
 * 상태 기록
 * - NORMAL : 입금 정상 처리
 * - ERROR  : 입금 처리 중 오류 발생
 */
```

### 3. 입금취소 (DEPOSIT_CANCEL) 처리 기준

```java
/**
 * 3. 입금취소 (DEPOSIT_CANCEL) 처리 기준
 *
 * [원거래 존재 + 상태 NORMAL]
 * - 정상적으로 완료된 입금 거래에 대한 취소 요청이다.
 * - 취소 대상 원거래가 NORMAL 상태이므로 입금 취소 프로세스를 수행한다.
 * - 일반적으로 취소 시작 시 상태를 CANCEL_PROC 로 변경한다.
 * - 이후 잔액 복구 및 거래 취소 기록 저장 등의 취소 로직을 수행한다.
 * - 처리 결과에 따라 상태를 CANCEL_NORMAL 또는 CANCEL_ERROR 로 갱신한다.
 *
 * [원거래 존재 + 상태 ERROR]
 * - 원거래 입금이 오류 상태로 종료된 경우이다.
 * - 실제 입금 처리가 수행되지 않았기 때문에 취소할 금액이 존재하지 않는다.
 * - 따라서 별도의 취소 처리를 수행하지 않는다.
 * - 기존 오류 상태를 기준으로 응답을 반환(REPLY_ORIGINAL)하거나
 *   취소 정상 응답을 반환하고 로그만 기록하는 방식으로 처리할 수 있다.
 *
 * [원거래 존재 + 상태 CANCEL_ERROR]
 * - 이전 입금 취소가 오류로 실패한 거래이다.
 * - 동일 취소 요청이 재수신된 경우 재처리를 허용한다.
 * - 네트워크 오류 또는 일시적 시스템 장애 등으로 취소가 완료되지 못한 경우
 *   정상 복구를 위해 취소 프로세스를 다시 수행한다.
 *
 * [원거래 존재 + 상태 PROCESSING]
 * - 원거래 입금이 아직 처리 중인 상태이다.
 * - 입금 처리가 완료되지 않은 상태에서는 취소 처리를 수행할 수 없다.
 * - 해당 취소 전문은 무시(DROP)하며 입금 처리 완료 후 재요청이 필요하다.
 *
 * [원거래 존재 + 상태 CANCEL_PROC]
 * - 동일 거래에 대해 이미 취소 처리 진행 중인 상태이다.
 * - 추가 취소 전문이 수신되면 중복 취소 또는 동시 처리 문제가 발생할 수 있다.
 * - 해당 전문은 무시(DROP)하고 기존 취소 프로세스 완료를 기다린다.
 *
 * [원거래 존재 + 상태 CANCEL_NORMAL]
 * - 이미 입금 취소가 정상 완료된 거래이다.
 * - 동일 취소 전문이 다시 수신되면 신규 취소 처리를 수행하지 않는다.
 * - 기존 취소 응답을 그대로 재응답(REPLY_ORIGINAL)하여 멱등성을 보장한다.
 *
 * [원거래 미존재]
 * - 취소 대상이 되는 입금 원거래가 시스템에 존재하지 않는다.
 * - 정상적인 금융 거래 흐름에서는 발생하면 안 되는 비정상 상황이다.
 * - 오류 응답(ERROR_NOT_FOUND)을 반환한다.
 * - 운영 로그 및 모니터링 대상 상황으로 관리한다.
 */
```

### 4. 출금 (WITHDRAW) 처리 기준

```java
/**
 * 4. 출금 (WITHDRAW) 처리 기준
 *
 * [원거래 존재]
 * - 동일 거래 식별자(거래번호, 전문번호 등)를 가진 출금 거래가 이미 시스템에 존재하는 경우이다.
 * - 출금은 실제 금액이 차감되는 금전 처리 트랜잭션이므로 동일 거래를 다시 처리하면
 *   중복 출금이 발생할 수 있다.
 * - 따라서 해당 전문은 중복 거래로 판단하고 즉시 무시(DROP)한다.
 * - 출금 결과가 잘못되었거나 정정이 필요한 경우에는 동일 출금 거래를 다시 처리하지 않는다.
 * - 정상적인 업무 절차에 따라 출금취소(WITHDRAW_CANCEL) 전문을 통해 정정하도록 한다.
 * - 이를 통해 거래 정합성 및 멱등성(Idempotency)을 유지한다.
 *
 * [원거래 미존재]
 * - 동일 거래 식별자가 시스템에 존재하지 않는 경우 신규 출금 요청으로 판단한다.
 * - 정상적인 출금 프로세스를 수행한다.
 *
 * 처리 절차
 * 1) 거래 정보 생성 및 상태 PROCESSING 기록
 * 2) 계좌 잔액 검증
 * 3) 출금 가능 여부 확인
 * 4) 금액 차감 처리
 * 5) 거래 기록 저장
 *
 * 상태 기록
 * - NORMAL : 출금 정상 처리
 * - ERROR  : 출금 처리 중 오류 발생
 */
```

### 5. 출금취소 (WITHDRAW_CANCEL) 처리 기준

```java
/**
 * 5. 출금취소 (WITHDRAW_CANCEL) 처리 기준
 *
 * [원거래 존재 + 상태 NORMAL]
 * - 정상적으로 완료된 출금 거래에 대한 취소 요청이다.
 * - 취소 대상 원거래가 NORMAL 상태이므로 출금 취소 프로세스를 수행한다.
 * - 취소 시작 시 상태를 CANCEL_PROC 로 변경한다.
 * - 이후 차감되었던 금액을 복구(잔액 환원)하고 취소 거래 기록을 저장한다.
 * - 처리 결과에 따라 상태를 CANCEL_NORMAL 또는 CANCEL_ERROR 로 갱신한다.
 *
 * [원거래 존재 + 상태 PROCESSING]
 * - 원거래 출금이 현재 처리 중인 상태이다.
 * - 출금 처리가 완료되지 않은 상태에서 취소가 수행되면 거래 정합성 문제가 발생할 수 있다.
 * - 해당 취소 전문은 무시(DROP)한다.
 * - 상태 변경 없이 기존 PROCESSING 상태를 유지한다.
 *
 * [원거래 존재 + 상태 CANCEL_PROC]
 * - 동일 거래에 대해 이미 출금 취소 처리 진행 중인 상태이다.
 * - 추가 취소 전문이 수신되면 중복 취소 또는 동시 처리 문제가 발생할 수 있다.
 * - 해당 전문은 무시(DROP)한다.
 * - 상태 변경 없이 기존 CANCEL_PROC 상태를 유지한다.
 *
 * [원거래 존재 + 상태 CANCEL_NORMAL]
 * - 이미 출금 취소가 정상 완료된 거래이다.
 * - 동일 취소 전문이 재수신되면 신규 취소 처리를 수행하지 않는다.
 * - 기존 취소 응답을 그대로 재응답(REPLY_ORIGINAL)하여 멱등성을 보장한다.
 * - 상태는 CANCEL_NORMAL 그대로 유지한다.
 *
 * [원거래 존재 + 상태 CANCEL_ERROR]
 * - 이전 출금 취소가 오류로 실패한 거래이다.
 * - 동일 취소 요청이 재수신된 경우 취소 재처리를 허용한다.
 * - 취소 재처리 시작 시 상태를 CANCEL_PROC 로 변경한다.
 * - 이후 처리 결과에 따라 CANCEL_NORMAL 또는 CANCEL_ERROR 로 갱신한다.
 *
 * [원거래 존재 + 상태 ERROR]
 * - 원거래 출금이 오류 상태로 종료된 경우이다.
 * - 실제 금액 차감이 발생하지 않았으므로 별도의 취소 처리가 필요하지 않다.
 * - 출금 취소 정상 응답을 반환한다.
 * - 취소 응답 코드 및 취소 이력만 기록하고 원거래 상태(ERROR)는 변경하지 않는다.
 *
 * [원거래 미존재]
 * - 취소 대상이 되는 출금 원거래가 시스템에 존재하지 않는 경우이다.
 * - 정상적인 거래 흐름에서는 발생하면 안 되는 상황이다.
 * - 다만 네트워크 지연 또는 메시지 순서 역전으로 인해
 *   출금취소 전문이 먼저 도착하고 이후 출금 원거래 전문이 도착할 수 있다.
 * - 이 경우 시스템은 출금취소 정상 응답을 반환한다.
 * - 취소 전문을 역거래(REVERSE) 형태로 저장한다.
 * - 이후 동일 거래번호의 출금 전문이 수신되면 해당 역거래 정보를 기반으로
 *   거래 정합성을 맞추도록 처리한다.
 * - 운영 모니터링 대상 상황으로 관리한다.
 *
 * ------------------------------------------------------------------
 * 상태 기록 정책 (Status Recording Policy)
 * ------------------------------------------------------------------
 *
 * 1. 상태 변경은 반드시 DB에 기록한다.
 *
 * 2. DROP 상황
 *    - 상태 변경 없이 로그만 기록한다.
 *
 * 3. 재처리 가능 상태 (Retryable)
 *    - CANCEL_ERROR
 *
 * 4. 멱등 상태 (Idempotent)
 *    - NORMAL
 *    - CANCEL_NORMAL
 *
 * 5. 진행 상태 (In Progress)
 *    - PROCESSING
 *    - CANCEL_PROC
 */
```

## 핵심 컴포넌트

### `KftcStatusEngine`

`(bizType, BizStatus)` 조합으로 `ProcessResult`를 결정하는 상태 머신입니다.

```java
@Component
public class KftcStatusEngine {

    public ProcessResult determineAction(String bizType, BizStatus currentStatus) {
        return switch (bizType) {
            case "INQUIRY"         -> (currentStatus != null) ? ProcessResult.DROP           : ProcessResult.PROCEED;
            case "DEPOSIT"         -> (currentStatus != null) ? ProcessResult.REPLY_ORIGINAL : ProcessResult.PROCEED;
            case "DEPOSIT_CANCEL"  -> handleDepositCancel(currentStatus);
            case "WITHDRAW_CANCEL" -> handleWithdrawCancel(currentStatus);
            default                -> ProcessResult.PROCEED;
        };
    }

    private ProcessResult handleDepositCancel(BizStatus status) {
        if (status == null) return ProcessResult.ERROR_NOT_FOUND;
        return switch (status) {
            case NORMAL                    -> ProcessResult.PROCEED;
            case CANCEL_ERROR, CANCEL_PROC -> ProcessResult.DROP;
            case CANCEL_NORMAL, PROCESSING -> ProcessResult.REPLY_ORIGINAL;
            default                        -> ProcessResult.DROP;
        };
    }
}
```

### `KftcTransactionService`

`KftcStatusEngine`이 결정한 `ProcessResult`에 따라 실제 처리를 분기합니다.

```java
@Service
@Transactional
public ResponseDto processTransaction(RequestDto request) {

    // 1. SELECT FOR UPDATE — 원거래 락 획득 (중복 처리 방지)
    TransactionEntity originTx = repository.findByTrxIdWithLock(request.getTrxId());

    // 2. 상태 판단
    BizStatus currentStatus = (originTx != null) ? originTx.getStatus() : null;
    ProcessResult action = statusEngine.determineAction(request.getBizType(), currentStatus);

    // 3. 결과별 분기
    return switch (action) {
        case PROCEED         -> executeBusinessLogic(request);        // 업무 로직 실행
        case DROP            -> ResponseDto.ignore();                  // 무응답 종료
        case REPLY_ORIGINAL  -> ResponseDto.fromEntity(originTx);     // 기존 결과 재전송
        case ERROR_NOT_FOUND -> ResponseDto.error("404", "원거래 없음");
        case SAVE_REVERSE    -> saveReverseTransaction(request);      // 역전 대비 저장
    };
}
```

```

    │
    ├─ SELECT FOR UPDATE (원거래 조회 + 행 레벨 락)
    │
    ├─ KftcStatusEngine.determineAction(bizType, currentStatus)
    │       │
    │       ├─ PROCEED         ──► 업무 로직 실행 → 원장 반영
    │       ├─ DROP            ──► 무시, 종료
    │       ├─ REPLY_ORIGINAL  ──► 원거래 응답 재전송 (원거래응답 보장)
    │       ├─ ERROR_NOT_FOUND ──► 404 오류 응답
    │       └─ SAVE_REVERSE    ──► 역전 거래 선행 저장
    │
    └─ ResponseDto 반환
```

## 설계 원칙

**원거래응답 (Idempotency)**
동일 전문이 재수신되면 원거래 응답을 그대로 반환합니다 (`REPLY_ORIGINAL`). 네트워크 재전송이나 타임아웃 재시도 상황에서 중복 처리를 방지합니다.

**역전 거래 방어 (Reverse Transaction Guard)**
출금취소 전문이 출금 원거래보다 먼저 도착한 경우, 취소 내역을 선행 저장합니다 (`SAVE_REVERSE`). 이후 출금 원거래가 도착했을 때 선저장된 취소 내역을 확인하여 정합성을 보장합니다.

**중복 전문 무시 (Duplicate Drop)**
이미 처리 중이거나 완료된 거래에 대한 재요청은 `DROP`으로 무시합니다. 취소 처리 중(`CANCEL_PROC`) 상태에서 동일 취소 전문이 재수신되는 경우를 포함합니다.

**비관적 락 (Pessimistic Locking)**
`SELECT FOR UPDATE`로 원거래 레코드에 행 레벨 락을 획득한 뒤 상태를 판단합니다. 동시 요청으로 인한 경쟁 조건(Race Condition)을 원천 차단합니다.
